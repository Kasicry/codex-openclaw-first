# IT News AI Briefing Agent Roadmap

## 1. 목표

여러 국내외 IT 뉴스 사이트에서 주요 뉴스를 수집하고, 필터링·요약·중복 제거를 거쳐 사용자가 하루 1회 실행만으로 5분 이내에 중요한 뉴스를 확인할 수 있는 시스템을 구축한다.

최종적으로 REST API와 OpenClaw를 연동하여 사용자가 자연어로 뉴스 브리핑을 조회할 수 있도록 확장한다.

## 2. 성공 기준

- 지원 사이트 중 3개 이상에서 뉴스를 안정적으로 수집한다.
- 한 번의 일일 브리핑 작업을 5분 이내에 완료한다.
- 한 사이트의 수집 실패가 다른 사이트의 수집을 중단시키지 않는다.
- 기사별 제목, 핵심 내용, 영향도, 개발자 관점 요약을 생성한다.
- 유사 기사를 통합하고 대표 기사만 제공한다.
- Telegram, REST API, OpenClaw 연동을 단계적으로 제공한다.
- 비밀정보를 저장소·로그·결과 파일에 기록하지 않는다.
- 외부 API 호출과 메시지 발송은 명시적 승인 및 설정 후에만 활성화한다.

## 3. 범위

### 포함

- 국내외 지정 뉴스 사이트 수집
- 키워드 기반 우선 필터링
- AI 기사 요약 및 중요도 분류
- 유사 기사 중복 제거
- 일별 JSON 저장
- Telegram 브리핑 발송
- REST API 제공
- OpenClaw 질의 연동

### 제외

- 댓글 및 SNS 분석
- 유료 뉴스 크롤링
- 실시간 스트리밍 수집
- 초기 버전의 SQLite/PostgreSQL 운영
- 초기 버전의 복잡한 관리자 UI

## 4. 기본 결정사항

- 초기 저장 형식은 `news/YYYY-MM-DD.json`으로 한다.
- 초기 구현은 단일 실행형 배치 작업을 우선한다.
- 프로젝트 소스와 문서는 `https://github.com/Kasicry/codex-openclaw-first.git`에서 버전 관리한다.
- 크롤링 구현은 `https://github.com/Kasicry/codex-crawling-starterkit.git`의 `main` 브랜치를 참고한다.
- 수집기는 사이트별로 실패를 격리한다.
- 우선 수집 키워드는 PRD에 정의된 키워드 목록을 사용한다.
- 중요도는 `높음`, `보통`, `낮음`의 3단계로 분류한다.
- 모든 외부 콘텐츠는 신뢰하지 않는 입력으로 취급한다.
- API 키, 토큰, 비밀번호는 환경 변수 또는 승인된 비밀 저장소에서만 주입한다.

## 5. 단계별 로드맵

일정은 1인 개발 기준의 목표 범위이며, 각 단계는 이전 단계의 완료 기준을 통과한 후 진행한다.

| 단계 | 목표 기간 | 핵심 결과 |
|---|---:|---|
| Phase 0. 기반 구축 | 1주 | 실행 가능한 프로젝트 골격과 데이터 계약 |
| Phase 1. 뉴스 수집 MVP | 2주 | 3개 이상 사이트 수집 및 JSON 저장 |
| Phase 2. AI 요약 | 1주 | 기사별 구조화 요약과 중요도 분류 |
| Phase 3. 중복 제거 | 1주 | 유사 기사 통합 및 대표 기사 선정 |
| Phase 4. Telegram 알림 | 1주 | 승인 기반 일일 브리핑 자동 발송 |
| Phase 5. REST API 연동 | 2주 | 최신·오늘·검색 API 제공 |
| Phase 6. OpenClaw 연동 | 2주 | 자연어 뉴스 질의 제공 |
| Phase 7. 운영 안정화 | 지속 | 품질, 성능, 관측성, 저장소 확장 검토 |

### Phase 0. 기반 구축

**목표**

수집부터 저장까지 공통으로 사용하는 최소 프로젝트 구조와 데이터 계약을 정의한다.

**주요 작업**

- `codex-crawling-starterkit`의 구조, 적용 가능 범위, 라이선스 검토
- 실행 환경, 언어, 패키지 관리 방식 확정
- 뉴스 기사와 브리핑 결과의 JSON 스키마 정의
- 사이트별 수집기 인터페이스와 실패 격리 방식 정의
- 환경 변수 예시 파일 작성 시 실제 비밀정보가 포함되지 않도록 구성
- 단위 테스트, 통합 테스트, 린트 실행 경로 마련
- 작업 시간과 사이트별 성공·실패를 기록하는 구조화 로그 정의

**산출물**

- 실행 가능한 프로젝트 골격
- 기사 및 브리핑 JSON 스키마
- 로컬 실행·테스트 문서
- 보안 체크리스트

**완료 기준**

- 샘플 기사 데이터를 입력하여 일별 JSON 결과를 생성할 수 있다.
- 비밀정보 없이 테스트와 린트를 실행할 수 있다.
- 사이트 하나의 실패를 격리하는 테스트가 통과한다.

### Phase 1. 뉴스 수집 MVP

**목표**

최소 3개 사이트에서 관련 뉴스를 수집하고 일별 JSON 파일로 저장한다.

**주요 작업**

- 우선 대상 3개 사이트 선정
- 사이트별 수집기 구현
- 제목, URL, 발행일, 출처, 본문 또는 요약 원문 수집
- PRD 키워드 기반 우선 필터링 구현
- URL 정규화와 기본 중복 제거 구현
- 타임아웃, 재시도, 사이트별 오류 격리 구현
- 전체 수집 시간 측정

**권장 우선 대상**

- OpenAI Blog
- Anthropic News
- Google AI Blog

RSS 또는 공식 피드가 제공되는 경우 HTML 크롤링보다 우선 사용한다.

**완료 기준**

- 3개 이상 사이트에서 기사를 수집한다.
- 결과가 `news/YYYY-MM-DD.json`에 저장된다.
- 특정 사이트 실패 시 나머지 사이트 수집이 계속된다.
- 정상적인 네트워크 조건에서 전체 작업이 5분 이내에 완료된다.
- 수집 성공·실패·소요 시간이 로그에 기록된다.

### Phase 2. AI 요약

**목표**

수집 기사마다 일관된 구조의 요약과 중요도를 생성한다.

**주요 작업**

- 요약 입력 길이 제한과 본문 정제 규칙 정의
- 외부 기사 내 지시문을 명령으로 취급하지 않는 프롬프트 구성
- 구조화 출력 스키마 적용
- 제목, 핵심 내용, 영향도, 개발자 관점 생성
- 중요도 `높음`, `보통`, `낮음` 분류
- API 실패, 속도 제한, 잘못된 출력에 대한 처리 구현
- 비용과 처리 시간 측정

**완료 기준**

- 수집된 기사에 구조화 요약이 생성된다.
- 출력 스키마 검증 실패 시 재시도하거나 안전하게 실패 처리한다.
- API 키와 기사 원문 내 개인정보가 로그에 노출되지 않는다.
- 샘플 기사 세트에 대한 요약 품질 검토가 완료된다.

### Phase 3. 중복 제거

**목표**

동일 사건을 다룬 유사 기사를 묶고 대표 기사만 브리핑에 포함한다.

**주요 작업**

- URL, 정규화 제목, 발행일 기반 1차 중복 제거
- 제목·요약 유사도 기반 2차 중복 제거
- 대표 기사 선정 규칙 정의
- 묶인 기사 출처 목록 보존
- 오탐과 누락을 확인할 평가 데이터셋 작성

**완료 기준**

- 동일 뉴스의 다중 게시 사례가 하나의 항목으로 통합된다.
- 대표 기사와 관련 출처를 확인할 수 있다.
- 평가 데이터셋 기준 오탐·누락 결과가 기록되고 허용 기준을 충족한다.

### Phase 4. Telegram 알림

**목표**

완성된 일일 브리핑을 Telegram으로 자동 발송한다.

**주요 작업**

- Telegram 메시지 템플릿 정의
- 메시지 길이 제한과 분할 발송 처리
- 발송 전 미리보기 및 수동 실행 경로 제공
- 발송 실패 재시도와 중복 발송 방지 구현
- 스케줄 실행 방식 결정

**보안 게이트**

- Telegram Bot Token과 대상 식별자는 저장소·로그에 기록하지 않는다.
- 실제 메시지 발송과 스케줄 활성화는 사용자 승인 후 수행한다.

**완료 기준**

- 승인된 테스트 채널에 요약을 정상 발송한다.
- 동일 브리핑이 중복 발송되지 않는다.
- 발송 실패가 수집·요약 결과를 손상시키지 않는다.

### Phase 5. REST API 연동

**목표**

외부 클라이언트가 저장된 뉴스를 조회할 수 있는 읽기 전용 REST API를 제공한다.

**주요 작업**

- API-LISTNER-STARTERKIT 연동 방식 검토
- 다음 조회 API 구현
  - `GET /news/latest`
  - `GET /news/today`
  - `GET /news/search`
- 검색어, 날짜, 중요도, 출처 필터 정의
- 입력 검증, 페이지네이션, 오류 응답 정의
- 읽기 전용 권한과 요청 제한 적용
- OpenAPI 명세와 계약 테스트 작성

**완료 기준**

- 세 가지 필수 조회 API가 동작한다.
- 잘못된 입력에 일관된 오류 응답을 반환한다.
- API를 통해 비밀정보나 내부 오류 상세가 노출되지 않는다.
- API 계약 테스트가 통과한다.

### Phase 6. OpenClaw 연동

**목표**

사용자가 자연어로 저장된 뉴스와 브리핑을 조회할 수 있도록 한다.

**주요 작업**

- OpenClaw에서 호출할 읽기 전용 도구 계약 정의
- 자연어 질의를 REST API 검색 조건으로 변환
- 날짜 범위, 주제, 출처 기반 질의 지원
- 응답에 기사 출처와 발행일 포함
- 프롬프트 인젝션과 권한 상승 방지 검증

**대표 사용자 시나리오**

- "오늘 AI 뉴스 요약"
- "이번주 OpenAI 뉴스"
- "Codex 관련 뉴스"

**완료 기준**

- 대표 사용자 시나리오 3개가 정상 동작한다.
- 답변에 근거 기사 링크와 발행일이 포함된다.
- OpenClaw 연동은 조회 전용 권한만 사용한다.
- 외부 문서의 지시가 시스템 명령이나 도구 호출로 실행되지 않는다.

### Phase 7. 운영 안정화 및 확장

**목표**

운영 품질을 높이고 데이터 증가에 따른 저장소 확장을 준비한다.

**주요 작업**

- 국내외 전체 대상 사이트로 수집 범위 확대
- 사이트별 성공률, 전체 소요 시간, 요약 실패율 모니터링
- 실패 알림과 재처리 절차 정의
- 비용, 품질, 성능 추세 점검
- JSON 저장 한계 도달 시 SQLite 전환 검토
- 다중 사용자·동시 요청 필요 시 PostgreSQL 전환 검토

**완료 기준**

- 운영 지표와 장애 대응 절차가 문서화된다.
- 저장소 전환 필요성을 데이터 규모와 조회 패턴으로 판단할 수 있다.
- 핵심 사용자 흐름에 대한 회귀 테스트가 자동화된다.

## 6. 공통 품질 게이트

각 Phase 완료 전 다음 항목을 확인한다.

- 기능 완료 기준과 자동화 테스트가 일치한다.
- 변경 범위에 대한 단위 또는 통합 테스트가 통과한다.
- 린트와 문서 일관성 검사를 통과한다.
- 한 구성요소 실패가 전체 작업을 불필요하게 중단시키지 않는다.
- 비밀정보와 개인정보가 코드, 로그, 결과 파일에 포함되지 않는다.
- 외부 API 호출과 메시지 발송은 승인된 경우에만 활성화된다.
- 운영 계정 대신 최소 권한의 조회 전용 계정을 사용한다.

## 8. 주요 리스크와 대응

| 리스크 | 영향 | 대응 |

| 사이트 구조 변경 또는 차단 | 수집 실패 | 공식 RSS/API 우선, 사이트별 수집기 격리, 실패율 모니터링 |
| AI 요약의 부정확성 | 브리핑 신뢰도 저하 | 출처 링크 제공, 구조화 출력 검증, 샘플 품질 평가 |

| 중복 제거 오탐 | 서로 다른 뉴스 누락 | 보수적 임계값, 관련 출처 보존, 평가 데이터셋 운영 |
| 외부 API 비용·속도 제한 | 처리 지연 또는 비용 증가 | 기사 수 제한, 재시도·백오프, 비용·시간 측정 |
| 비밀정보 노출 | 보안 사고 | 환경 변수 사용, 로그 마스킹, 저장소 검사 |
